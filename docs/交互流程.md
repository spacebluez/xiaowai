## 2. 首页流程

首页包含两个主要按钮：

### 2.1 开始使用
前端点击“开始使用”后，先校验当前登录状态。

#### 推荐流程
1. 前端请求 `GET /api/v1/user/profile`
2. 如果返回成功，说明用户已登录，跳转到聊天页或主工作台
3. 如果返回 `401`，说明未登录或登录失效，跳转到登录页

> 当前前端现状：`Home.tsx` 中“开始使用/开始对话”是直接跳转页面；联调时建议改为先校验登录态，再决定跳转目标。

### 2.2 了解更多
当前可跳转到产品介绍页、登录页或待定页面。

## 3. 注册流程

当前后端注册采用“邮箱验证码 + 用户名 + 密码”的流程。

### 3.1 页面字段
注册页建议包含以下字段：

- 用户名 `username`
- 邮箱 `email`
- 验证码 `code`
- 密码 `password`
- 确认密码 `confirmPassword`
- 手机号 `phone`（可选）
- 用户协议勾选框

### 3.2 获取验证码流程
1. 用户输入邮箱
2. 点击“获取验证码”按钮
3. 前端调用 `POST /api/v1/verification/send-code`
4. 后端发送邮件验证码
5. 前端开始 60 秒倒计时，倒计时内不可重复发送

### 3.3 提交注册流程
1. 用户填写验证码、用户名、密码、手机号（可选）
2. 前端校验：
   - 邮箱格式合法
   - 验证码为 6 位
   - 用户名长度 2 到 50
   - 密码长度至少 6 位
   - 确认密码一致
3. 前端调用 `POST /api/v1/user/register`
4. 注册成功后跳转登录页
5. 注册失败则展示后端返回错误信息

### 3.4 注册失败提示建议
- 用户名已存在
- 邮箱已存在
- 验证码错误
- 验证码已过期
- 参数错误
- 系统异常，请稍后重试

### 3.5 与当前前端代码差异
当前 `Register.tsx` 只有：
- 用户名
- 邮箱
- 密码
- 确认密码

联调时需要补充：
- 验证码输入框
- 获取验证码按钮
- 手机号输入框（可选）
- 调用后端真实接口

## 4. 登录流程

### 4.1 页面字段
登录页建议使用：

- 用户名 `username`
- 密码 `password`

> 注意：后端登录接口使用的是用户名，不是邮箱。
> 当前 `Login.tsx` 页面展示的是“邮箱地址”，联调时应改为“用户名”或改成“用户名/邮箱”并同步后端。

### 4.2 登录交互流程
1. 用户输入用户名和密码
2. 前端调用 `POST /api/v1/user/login`
3. 后端校验账号密码
4. 登录成功后：
   - 响应体返回 token
   - 同时通过 Cookie 写入 `token`
5. 前端跳转聊天页或主页面
6. 页面初始化时调用 `GET /api/v1/user/profile` 拉取用户资料

### 4.3 登录失败处理
- 用户名或密码错误：提示“用户名或密码错误”
- 连续失败过多：提示“登录失败次数过多，请10分钟后再试”
- 其他错误：提示“登录失败，请稍后重试”

### 4.4 登录态保持
后端认证依赖 Cookie，因此前端后续所有需要登录的请求都必须携带：

- `fetch`：`credentials: 'include'`
- `axios`：`withCredentials: true`

## 5. 用户资料流程

### 5.1 进入个人资料页
1. 前端请求 `GET /api/v1/user/profile`
2. 将返回数据填充到资料表单中
3. 若返回 `401`，跳转登录页

### 5.2 资料字段
后端支持以下资料字段：

- 昵称 `nickname`
- 生日 `birthday`，格式 `YYYY-MM-DD`
- 头像 URL `avatar`
- 性别 `gender`，取值：`unknown | male | female | other`
- 爱好 `hobbies`
- 个性签名 `signature`

### 5.3 更新资料流程
1. 用户编辑资料
2. 前端调用 `PUT /api/v1/user/profile`
3. 后端返回最新资料
4. 前端用最新资料刷新页面

### 5.4 上传头像流程
1. 用户选择图片文件
2. 前端构造 `FormData`
3. 调用 `PUT /api/v1/user/avatar`
4. 上传成功后重新拉取用户资料，或直接刷新头像显示

### 5.5 头像展示流程
前端可使用以下地址作为头像展示地址：

`GET /api/v1/user/:id/avatar`

说明：
- 这是公开接口，不要求登录
- 成功时返回图片流，不是 JSON
- 如果用户没有有效头像，前端应显示默认占位头像

### 5.6 与当前前端代码差异
当前 `Profile.tsx` 为本地 mock 数据，联调时需替换为：
- 页面加载请求真实资料
- 保存按钮调用更新资料接口
- 头像点击上传调用头像接口
- 邮箱、手机号等字段需与后端返回结构重新映射

## 6. 聊天流程

### 6.1 进入聊天页
1. 前端先校验登录状态
2. 如果未登录，跳转登录页
3. 如果已登录，进入聊天页

### 6.2 发送消息流程
1. 用户输入文本
2. 前端将用户消息先插入本地消息列表
3. 调用 `POST /api/v1/user/agent/chat`
4. 后端返回 AI 回复
5. 前端将 AI 回复追加到消息列表

### 6.3 错误处理
- 未登录：跳转登录页
- 参数错误：提示“请输入消息内容”
- 服务异常：提示“对话失败，请稍后再试”

### 6.4 当前能力边界
当前后端聊天接口特点：

- 仅支持文本输入
- 当前是单轮问答接口
- 没有历史记录存储接口
- 没有多轮上下文记忆接口
- 没有智能体列表接口

因此前端当前页面中的以下内容暂时属于展示态或 mock：
- 最近对话
- 智能体切换
- 历史记录
- Prompt 广场
- 设置中与后端联动的高级能力

---

## 7. 退出登录流程

1. 用户点击“退出登录”
2. 前端调用 `POST /api/v1/user/logout`
3. 后端清除 Cookie 中的 `token`
4. 前端清空本地用户状态
5. 跳转首页或登录页

## 9. 联调注意事项

### 9.1 认证方式
虽然登录接口会返回 `token`，但后端真正鉴权依赖 Cookie，因此必须带凭证请求。

### 9.2 响应格式
大部分接口返回：

```json
{
  "code": 0,
  "msg": "ok",
  "data": {}
}
```

但鉴权中间件失败时可能直接返回：

```json
{
  "msg": "请先登录"
}
```

前端要兼容两种错误格式。

### 9.3 CORS 来源限制
当前后端只允许以下前端来源跨域：

- `http://localhost:5173`
- `http://47.111.12.222:5173`

如果前端运行端口或域名变化，需要同步修改后端 CORS 配置。

### 9.4 默认头像问题
后端注册时默认头像是字符串 `default_avatar.png`，但头像预览接口预期是 `oss://...` 路径。联调时若头像加载失败，前端应先显示默认占位图。

---

## 10. 推荐的实际联调顺序

1. 打通验证码发送接口
2. 打通注册接口
3. 打通登录接口
4. 打通登录态校验接口 `GET /user/profile`
5. 打通资料更新接口
6. 打通头像上传与头像预览接口
7. 打通聊天接口
8. 最后再接历史记录、智能体管理等尚未提供的后端能力
